# 给 Codex 的交接说明

这份 prototype 是一个“毛坯房”：已经有笔记、文件夹、搜索、标签、摘要视图和导入导出，但没有绑定任何私人代码、私人数据、API key、数据库或部署环境。你可以把下面的 prompt 直接复制给 Codex，让它按阶段帮你搭起来。

## 最开始直接复制这段

```text
我在做一个个人 Workstation / 知识库产品，现在还是很早期。

请先阅读当前文件夹里的代码，不要接入任何真实 API，不要添加密钥，不要部署，也不要把复杂后端一次性做完。

我的目标是先把这个本地 prototype 变成一个好用的笔记工作台：
- 左边有文件夹和标签
- 中间是笔记列表
- 右边是编辑器
- 支持搜索、置顶、导入导出
- 后续可能接 Supabase、Cloudflare 和 AI，但现在先保持简单

请你先做三件事：
1. 总结当前代码结构和它是怎么运行的。
2. 给我一个适合新手的下一步开发计划。
3. 只实现一个小改动，并告诉我怎么验证它。
```

## 每次让 Codex 开工前都加这段

```text
请保持改动小而清楚。不要重写整个项目。不要把 API key 写进前端代码。任何部署、数据库迁移、删除数据、推送到 main/master 之前都先问我。
```

## 适合第一周做的 prompt

```text
帮我把笔记编辑体验做得更好：
- 支持 Markdown 预览
- 编辑和预览可以切换
- 不要引入复杂后端
- 做完后告诉我改了哪些文件，以及怎么测试
```

```text
帮我增加“最近更新”视图：
- 可以看到最近 7 天改过的笔记
- 笔记列表按更新时间排序
- 左侧增加一个入口
- 保持界面简洁
```

```text
帮我增加“公司 / 项目 / 人物”字段：
- 每条笔记可以关联一个或多个实体
- 先存在 localStorage
- 搜索时也能搜到这些实体
- 不要接数据库
```

## 准备接 Supabase 时用

```text
我准备把这个本地 prototype 升级到 Supabase。

请先不要直接改代码。先给我一个方案：
- 哪些数据表需要建
- 每张表有哪些字段
- 哪些表需要 Row Level Security
- 前端哪些地方要从 localStorage 改成 Supabase
- API key 应该放在哪里，哪些绝对不能放进浏览器

请参考当前 Supabase 官方文档再回答。方案确认后再开始实现。
```

## 准备接 Cloudflare 时用

```text
我准备把这个 Workstation prototype 部署到 Cloudflare。

请先判断更适合 Cloudflare Pages 还是 Workers。
不要直接部署。先告诉我：
- 推荐架构
- 需要哪些环境变量
- 哪些密钥只能放在服务端
- 本地如何测试
- 部署前检查清单
```

## 准备接 AI 时用

```text
我想给这个 Workstation 加 AI 功能，但还没决定用哪个模型。

请先设计一个 provider-agnostic 的方案：
- 前端只发“生成摘要 / 提取行动项 / 问答”的请求
- 后端负责调用模型
- API key 只放服务端环境变量
- 先支持一个 mock provider，之后再接真实模型
- 不要把模型供应商写死在 UI 里
```

## 如果 Codex 一次改太多

```text
先停一下。请把这次任务缩小成一个 1 小时内能完成的小改动，只改必要文件，并先解释你准备改什么。
```

## 如果代码报错

```text
请你先复现错误，然后定位原因。不要靠猜。修复后请运行验证，并用简单语言告诉我：
- 错误是什么
- 为什么会发生
- 你改了什么
- 我怎么确认已经好了
```

## 推荐开发顺序

1. 先把本地笔记体验做顺。
2. 再加 Markdown、最近更新、实体关联。
3. 再加 Supabase 登录和数据库。
4. 再加 Cloudflare 部署。
5. 最后再接真实 AI 模型。

早期最重要的是：每次只做一个清楚的小功能，能运行、能验证、能回滚。
